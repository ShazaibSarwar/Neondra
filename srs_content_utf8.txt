This Software Requirements Specification (SRS) defines the complete functional and non-functional requirements for the Wedding Financial & Gift Tracking System (WFGTS). 2. Executive Summary
The Wedding Financial & Gift Tracking System (WFGTS) is a mobile-first web application purpose-built for Pakistani families to digitize and manage the complex financial interactions that occur during multi-event wedding ceremonies. In traditional Pakistani weddings, monetary gifts (Salami) and contributions flow between dozens of families across four or more ceremonial events — Mehndi, Barat, Valima, and Nikah — yet all record-keeping is currently done informally or from memory.
WFGTS solves this by providing a structured digital platform where families can create wedding events, register participants, record every transaction in real time, and instantly view net balances and summaries. The system is designed for one-handed mobile use during live events, with minimal typing, fast data entry flows, and card-based dashboards.
2.1 Key Value Proposition
Eliminates manual record-keeping and memory-based tracking
Provides real-time net balance calculations per person and per event
Supports the full Pakistani wedding event structure (Mehndi, Barat, Valima, Nikah)
Offers role-based family management for multi-member coordination
Generates exportable PDF and Excel reports for post-event reconciliation
2.2 Success Criteria

5. Functional Requirements
5.1 FR-AUTH — Authentication & User Management
This module governs all identity and access management operations for the WFGTS platform.
FR-AUTH-001  |  User Registration  |  Priority: MUST HAVE
Description: The system shall allow new users to register with a full name, email address, and password. Upon submission, the system validates email uniqueness, enforces password strength requirements (minimum 8 characters, at least one uppercase letter, one digit), and sends a verification email before activating the account. Acceptance Criteria: User can register with valid credentials. Duplicate email shows error. Weak password shows validation message. Unverified account cannot log in.

FR-AUTH-002  |  User Login  |  Priority: MUST HAVE
Description: The system shall authenticate registered users via email and password. On successful login, a JWT access token (expiry: 15 minutes) and refresh token (expiry: 7 days) are issued and stored securely. Failed login increments an attempt counter; after 5 consecutive failures, the account is temporarily locked for 15 minutes. Acceptance Criteria: Valid credentials return JWT tokens. Invalid credentials show error. 5th failed attempt triggers lockout message. Locked account cannot log in until timeout expires.

FR-AUTH-003  |  JWT Token Refresh  |  Priority: MUST HAVE
Description: The system shall automatically refresh the access token using the refresh token before expiry. If the refresh token is expired or invalid, the user is redirected to the login screen. Token rotation must be implemented — each refresh issues a new refresh token and invalidates the old one. Acceptance Criteria: Access token is refreshed silently. Expired refresh token redirects to login. Reused refresh tokens are rejected.

FR-AUTH-004  |  Password Reset  |  Priority: MUST HAVE
Description: The system shall provide a password reset flow initiated by email. The system sends a time-limited (60-minute) reset link. On visiting the link, the user enters and confirms a new password. Used reset links are invalidated immediately. Acceptance Criteria: Valid email triggers reset email within 30 seconds. Expired link shows expiry error. Successful reset allows login with new password. Old password no longer works.

FR-AUTH-005  |  User Profile Management  |  Priority: SHOULD HAVE
Description: The system shall allow authenticated users to update their display name, profile photo (max 2MB, JPEG/PNG), and phone number. Email address changes require re-verification. Password changes require current password confirmation. Acceptance Criteria: Profile updates persist immediately. Invalid photo format/size shows error. Email change triggers verification email. Password change without correct current password is rejected.

FR-AUTH-006  |  Logout & Session Termination  |  Priority: MUST HAVE
Description: The system shall invalidate both access and refresh tokens on logout, add them to a server-side deny list, and clear all client-side stored credentials. Acceptance Criteria: Post-logout API calls with old tokens return 401. User is redirected to login page. Browser storage is cleared.

5.2 FR-FAM — Family Management
Families are the top-level organizational unit in WFGTS. All weddings, persons, and transactions belong to a family group.
FR-FAM-001  |  Create Family  |  Priority: MUST HAVE
Description: An authenticated user can create a family group by providing a family name (required, 2–100 characters), optional description, and optional family photo. The creating user automatically becomes the Family Admin. Acceptance Criteria: Family is created and appears in the user's family list. Creator is assigned Admin role. Duplicate family names under the same user are allowed but warned.

FR-FAM-002  |  Invite Family Members  |  Priority: MUST HAVE
Description: A Family Admin can invite other registered users to join the family by email. The invitee receives an email with an accept/decline link. The inviting admin assigns a role (Admin, Member, or Viewer) at invitation time. Acceptance Criteria: Invitation email is sent within 60 seconds. Accepting the invitation adds the user to the family with assigned role. Declining removes the pending invitation.

FR-FAM-003  |  Role-Based Access Control  |  Priority: MUST HAVE
Description: The system shall enforce three roles within a family — Admin (full CRUD on all family data), Member (can add/edit transactions; cannot delete family or members), and Viewer (read-only access). Permission checks are enforced at the API level, not just UI level. Acceptance Criteria: Member attempting delete operation receives 403 error. Viewer attempting any write operation receives 403 error. Admin can perform all operations.

FR-FAM-004  |  Manage Family Members  |  Priority: MUST HAVE
Description: A Family Admin can view all members, change their roles, and remove them from the family. An admin cannot remove themselves if they are the sole admin — the system must enforce at least one admin per family. Acceptance Criteria: Role change takes effect immediately. Removing a member revokes their access. Removing the last admin is blocked with an explanatory error message.

FR-FAM-005  |  Add Person Profile  |  Priority: MUST HAVE
Description: Family Admins and Members can add person profiles (guests, relatives) to the family. Each person profile includes: full name (required), relation type (e.g., Uncle, Aunt, Family Friend — configurable list), optional phone number, optional notes, and optional profile photo. Acceptance Criteria: Person is saved and appears in the family's contact list. Duplicate names in the same family show a warning (not a hard block). Person profiles are searchable by name.

5.3 FR-WED — Wedding Event Management
Each family can manage multiple weddings. A wedding is a parent entity that groups one or more ceremonial sub-events.
FR-WED-001  |  Create Wedding  |  Priority: MUST HAVE
Description: A Family Admin or Member can create a wedding record with: wedding title (required), groom name, bride name, wedding date (required), wedding location (city/venue), linked family (required), and status (upcoming, active, completed). The system supports multiple weddings per family. Acceptance Criteria: Wedding record is saved and appears in the family dashboard. Required fields validated before save. Wedding date cannot be in the past when creating an 'upcoming' record.

FR-WED-002  |  Manage Wedding Events (Ceremonies)  |  Priority: MUST HAVE
Description: Each wedding can have multiple ceremony events. Supported types are: Mehndi, Barat, Valima, Nikah, and Other (with custom label). Each ceremony event has: event type (required), event date (required), venue, start time, and notes. Multiple events of the same type are allowed (e.g., two Mehndi events). Acceptance Criteria: Ceremony events are saved under the parent wedding. All five event types are selectable. Custom label is required when 'Other' is selected. Events are listed in chronological order.

FR-WED-003  |  Edit & Archive Wedding  |  Priority: MUST HAVE
Description: Family Admins can edit all wedding fields. Archiving a wedding moves it to a read-only archive state; no new transactions can be added to archived weddings. Archived weddings are accessible for reporting. Acceptance Criteria: Edits persist immediately. Archived weddings appear in a separate archive list. Attempting to add a transaction to an archived wedding returns an error.

5.4 FR-TXN — Financial Transaction Tracking
Transactions are the core data entity of WFGTS. Every financial exchange is recorded as a transaction linked to a person, an event, and an amount.
FR-TXN-001  |  Record Cash Transaction (Salami)  |  Priority: MUST HAVE
Description: A Member or Admin can record a cash transaction with: sender (required — selected from person list or typed as free text for new entries), receiver (required), amount in PKR (required, minimum 1, maximum 10,000,000), associated ceremony event (required), transaction date (required, defaults to today), transaction type (Cash Gift / Salami), and optional note (max 200 characters). The system saves the transaction and immediately updates all balance calculations. Acceptance Criteria: Transaction is saved with all required fields. Amount field accepts only numeric input. Balance calculations update instantly after save. Transaction appears in the event's transaction list.

FR-TXN-002  |  Record Gift Item Transaction  |  Priority: SHOULD HAVE
Description: A Member or Admin can record a gift item transaction with: item description (required), estimated monetary value (optional), sender, receiver, ceremony event, and date. Gift items appear in the transaction list with a distinct visual indicator and are optionally included in balance calculations based on a user preference toggle. Acceptance Criteria: Gift items are saved and displayed with a gift icon. If value is provided and the 'include gifts in balance' toggle is on, the value is added to the sender's total given. If toggle is off, gift items appear in history but do not affect numeric balances.

FR-TXN-003  |  Quick Transaction Entry (Fast Mode)  |  Priority: MUST HAVE
Description: The system shall provide a 'Quick Add' mode accessible via a floating action button on mobile. Quick Add pre-fills the current date and the most recently used event, requiring only sender, receiver, and amount to complete an entry. The form shall be completable in under 10 seconds on a mobile device. Acceptance Criteria: Quick Add form opens in a bottom sheet. Three required fields are visible without scrolling. Submit saves the transaction in under 1 second. Success confirmation is shown for 2 seconds then auto-dismisses.

FR-TXN-004  |  Edit Transaction  |  Priority: MUST HAVE
Description: Any transaction can be edited by an Admin or the Member who created it. Edits are timestamped and the last-modified user is tracked. Editing amount or sender/receiver triggers recalculation of affected balances. Acceptance Criteria: Edited fields are saved correctly. Balance recalculation completes within 1 second. Last modified timestamp and user are visible in the transaction detail view.

FR-TXN-005  |  Delete Transaction  |  Priority: MUST HAVE
Description: Admins can delete any transaction. Members can delete only their own entries within 24 hours of creation. Deletion is soft (marked deleted, not removed from DB) to enable audit trails. Deleted transactions are removed from all balance calculations immediately. Acceptance Criteria: Admin delete succeeds for any transaction. Member delete succeeds only for own transactions within 24 hours. Deleted transactions do not appear in lists or balances. Soft-deleted records are retained in the audit log.

FR-TXN-006  |  Bulk Transaction Import  |  Priority: COULD HAVE
Description: Admins can import transactions from a CSV template. The system provides a downloadable CSV template. On upload, the system validates each row, reports validation errors per row, and processes valid rows. Import is atomic per batch — if >20% of rows fail validation, the entire batch is rejected. Acceptance Criteria: Template download works correctly. Valid CSV imports all rows. Row-level errors are displayed with line numbers. Batch rejection threshold enforced. Successfully imported transactions appear in the transaction list.

5.5 FR-BAL — Balance Calculation System
FR-BAL-001  |  Per-Person Balance Calculation  |  Priority: MUST HAVE
Description: The system shall automatically maintain a running balance for each person: Total Given (sum of all transactions where the person is the sender), Total Received (sum of all transactions where the person is the receiver), and Net Balance (Total Given minus Total Received). Balances are updated in real time upon any transaction create, update, or delete. Acceptance Criteria: All three balance figures are accurate to the rupee. Balances update within 1 second of any transaction change. Zero balance persons are displayed distinctly. Negative balances (received more than given) are highlighted in red.

FR-BAL-002  |  Event-Wise Balance Breakdown  |  Priority: MUST HAVE
Description: The system shall provide a balance breakdown per ceremony event: total collected during the event, total distributed, and net event balance. Each event's breakdown is accessible from the event detail screen. Acceptance Criteria: Event totals match the sum of all transactions for that event. Breakdown is visible within 2 taps from the main dashboard.

FR-BAL-003  |  Family-Wide Summary  |  Priority: MUST HAVE
Description: The system shall display a family-level financial summary across all weddings: grand total given by the family, grand total received, overall net balance, and a ranked list of top contributors. Acceptance Criteria: Family summary values match the sum of all individual balances. Top contributors list is sorted by total given, descending. Summary is visible on the main dashboard.

FR-BAL-004  |  Cross-Wedding Balance Tracking  |  Priority: SHOULD HAVE
Description: For persons who appear across multiple weddings (e.g., a relative who gave at both Barat and Valima across two different family weddings), the system shall provide a consolidated view showing their total contributions across all linked weddings. Acceptance Criteria: Cross-wedding totals are accurate. Person appearing in multiple weddings has a consolidated profile view. Data is de-duplicated based on person ID, not name matching.

5.6 FR-DASH — Dashboard & Analytics
FR-DASH-001  |  Family Overview Dashboard  |  Priority: MUST HAVE
Description: The main dashboard shall display: family name and active wedding count, a list of wedding cards (title, date, total transactions, total amount), quick-access buttons for Add Transaction and View Reports, and notification badges for recent activity. The dashboard loads within 2 seconds on a 4G mobile connection. Acceptance Criteria: All dashboard elements render correctly. Wedding cards show accurate counts and amounts. Dashboard loads within 2 seconds. Quick-access buttons navigate to correct screens.

FR-DASH-002  |  Wedding Summary Card  |  Priority: MUST HAVE
Description: Each wedding has a summary card showing: wedding title, bride/groom names, date, total events, total transactions count, total amount collected, and a progress indicator for events completed vs. total events. Acceptance Criteria: All fields display correct data. Progress indicator reflects actual event status. Tapping the card navigates to the wedding detail screen.

FR-DASH-003  |  Analytics Charts  |  Priority: SHOULD HAVE
Description: The system shall provide visual analytics including: (1) Bar chart comparing total amounts per ceremony event, (2) Pie chart showing contribution distribution among top contributors, (3) Line chart showing transaction volume over time. Charts are rendered using a lightweight library and are interactive (tap to see values). Acceptance Criteria: All three chart types render without errors. Charts show correct data. Interactive tooltips display on tap. Charts are readable on a 5-inch mobile screen.

FR-DASH-004  |  Individual Contribution History  |  Priority: MUST HAVE
Description: Tapping on any person profile opens their contribution history: a chronological list of all transactions they sent or received, their total given, total received, net balance, and a per-event breakdown. The history can be filtered by event and date range. Acceptance Criteria: History list is complete and accurate. Filters work correctly. Net balance matches the sum of listed transactions.

5.7 FR-SRCH — Search & Filter
FR-SRCH-001  |  Global Person Search  |  Priority: MUST HAVE
Description: The system shall provide a real-time search bar that filters persons by name across the current family. Search results appear as the user types (debounced at 300ms). Search is case-insensitive and supports partial matches. Acceptance Criteria: Results appear within 300ms of typing. Partial name matches are returned. No results found state is shown when no matches exist.

FR-SRCH-002  |  Transaction Filter Panel  |  Priority: MUST HAVE
Description: The transaction list shall include a filter panel with options: Filter by Event (multi-select checklist of events), Filter by Person (sender or receiver), Filter by Date Range (start date, end date), Filter by Type (Cash / Gift Item). Active filters are indicated by a badge count on the filter button. Acceptance Criteria: All filter options function correctly. Combined filters work (AND logic). Active filter count badge updates. Clearing filters restores the full list.

5.8 FR-RPT — Reports & Export
FR-RPT-001  |  PDF Export  |  Priority: MUST HAVE
Description: The system shall generate a formatted PDF report for a selected wedding or event. The report includes: cover page with wedding details, transaction log sorted by event and date, per-person balance summary table, event-wise totals, and a grand total. The PDF is formatted for A4 paper and optimized for printing. Acceptance Criteria: PDF generates within 3 seconds. All data in the PDF matches on-screen data. PDF is downloadable on mobile and desktop. Table formatting is correct on A4 paper.

FR-RPT-002  |  Excel Export  |  Priority: MUST HAVE
Description: The system shall export transaction data to an Excel (.xlsx) file with three worksheets: (1) All Transactions, (2) Person Summary, (3) Event Summary. Each sheet includes appropriate column headers and auto-fitted column widths. Acceptance Criteria: Excel file downloads successfully. All three sheets are present with correct data. Column headers match field names. Data is accurate to the last transaction.

FR-RPT-003  |  Printable Event Summary  |  Priority: SHOULD HAVE
Description: The system shall generate a print-optimized one-page summary for a single ceremony event, showing all transactions for that event in a clean table format suitable for quick reference during or after the event. Acceptance Criteria: Print view is triggered by browser print. Layout fits on a single A4 page for up to 50 transactions. Page breaks handle overflow correctly.

8. API Specification
The WFGTS backend exposes a RESTful JSON API. All endpoints require Bearer token authentication unless marked Public. Base URL: /api/v1

11. Business Rules
ID | Rule | Enforcement Level
BR-001 | A transaction amount must be a positive number greater than zero and no more than PKR 10,000,000 | Backend API + Frontend validation
BR-002 | A sender and receiver in the same transaction must be different persons | Backend API + Frontend validation
BR-003 | Transactions can only be added to ceremony events belonging to the user's own family | Backend API (family scope check)
BR-004 | A family must always have at least one Admin — the last Admin cannot be removed or demoted | Backend API
BR-005 | Archived weddings are read-only — no transactions, events, or edits allowed | Backend API
BR-006 | Members can only delete their own transactions within 24 hours of creation | Backend API (timestamp + ownership check)
BR-007 | The 'Other' ceremony event type requires a custom label of at least 2 characters | Backend API + Frontend validation
BR-008 | Gift items with no monetary value do not affect balance calculations regardless of toggle setting | Backend API
BR-009 | A person profile cannot be deleted if they have associated transactions; they can only be archived | Backend API (FK constraint + business rule)
BR-010 | Invitation links expire after 72 hours; expired invitations must be re-sent | Backend API (timestamp check)
BR-011 | Balance calculations always exclude soft-deleted transactions | Backend API query filter
BR-012 | All monetary amounts are stored and displayed in Pakistani Rupees (PKR); no currency conversion | System-wide

14. Glossary
Term | Definition
Salami | Urdu/Punjabi term for cash gifts traditionally given during Pakistani wedding ceremonies
Barat | The wedding procession ceremony — the primary wedding event in Pakistani culture
Mehndi | Pre-wedding ceremony involving henna application and celebrations, typically 1–2 days before Barat
Valima | Post-wedding reception hosted by the groom's family, typically the day after Barat
Nikah | The Islamic marriage contract ceremony; may be held separately from Barat festivities
Net Balance | Total amount given by a person minus total amount received; positive = net giver, negative = net receiver
Ceremony Event | A single ceremonial occasion within a wedding (e.g., one Mehndi event on a specific date)
Family | A top-level organizational group in WFGTS representing one nuclear or extended family unit
Person Profile | A registered contact record representing a guest, relative, or participant tracked in the system
Soft Delete | A deletion approach where records are marked as deleted (is_deleted = true) but retained in the database for audit purposes
JWT | JSON Web Token — a compact, URL-safe means of representing authentication claims between parties
SRS | Software Requirements Specification — this document
WFGTS | Wedding Financial & Gift Tracking System — the product described in this document
FAB | Floating Action Button — a circular button that hovers above screen content, triggering the primary screen action
RBAC | Role-Based Access Control — the permission model where access rights are assigned to roles (Admin, Member, Viewer)
PKR | Pakistani Rupee — the currency unit used throughout the system
